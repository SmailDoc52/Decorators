## Декораторы (`tools/decorators.py`)

Содержит два типа логгеров, которые автоматически фиксируют:

* Дату и время вызова функции;

* Имя функции;

* Переданные аргументы (`args`, `kwargs`);

* Возвращаемое значение.

* `logger`: Записывает логи в фиксированный файл `main.log`.

* `logger_with_path(path)`: Принимает путь к файлу в качестве аргумента.


## Менеджер данных (`tools/csv_manager.py`)

Класс `CSVManager` инкапсулирует логику чтения и записи CSV-файлов.

* Статический метод `cleaned_phone` производит стандартизацию телефонных номеров с помощью регулярных выражений (`re`).

* Метод обернут декоратором `@logger('csv_manager.log')` для автоматического отслеживания процесса очистки данных.

## Моя личная песочница (Эксперимент в logger_4.py)

В ходе работы над проектом был написан дополнительный скрипт `logger_4.py` (совместно с модулем `tools/open_file.py`). 

```bash
### В чем была идея:

Обычно файл открывается и закрывается каждый раз, когда функция пишет лог. Я подумал: "Зачем постоянно открывать и закрывать файл?" Идея заключалась в том, чтобы открывать файл для записи лога **ровно один раз** при инициализации декоратора (используя генератор с `yield`), вместо постоянного открытия/закрытия файла через менеджер контекста `with open()` при каждом вызове функции.

### Почему это оказалось плохой идеей на практике:
**Операционная система кэшировала логи** в оперативной памяти и не сбрасывала их на жесткий диск мгновенно. Для корректного прохождения тестов на чтение файлов пришлось принудительно вызывать file.flush(), что нивелировало потенциальную выгоду от «быстрого» удержания потока.

### Итог:
Эксперимент провалился, зато я на практике поработал с генератором и смог удержать файл открытым. В итоге для домашки я оставил классический и надежный вариант с `with open()`. А файл `logger_4.py` оставил в репозитории как пример выполнения работы.
```
